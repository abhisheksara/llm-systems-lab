Autonomous Research Harnesses,
Taught Through AutoResearch

Karpathy's autoresearch is 1,000 lines that let an AI agent improve a model overnight. Stripped of the GPU specifics, it's a reusable blueprint for any "propose → run → measure → keep-or-discard" loop. This course teaches the blueprint — then shows you how to point it at your own research problem.

~45 min read Repo: ../autoresearch/ Goal: design + run your own loop
Part I

The Big Idea

What AutoResearch actually is

AutoResearch (Andrej Karpathy, March 2026) is a deliberately tiny harness that hands an AI coding agent a real-but-small LLM training setup and tells it: experiment on your own, overnight, and only keep what measurably helps. You describe research directions in plain English, point a coding agent (Claude Code, Codex, etc.) at the repo, disable its permission prompts, and walk away. You wake up to a log of ~100 experiments and, hopefully, a better model.

The headline result: on a setup Karpathy had already hand-tuned over years, the agent ran ~700 changes autonomously, found ~20 that genuinely helped, and stacking them gave an ~11% efficiency gain. Shopify's CEO adapted the same pattern overnight and got a 0.8B model beating his hand-tuned 1.6B baseline.

The point of this course

You are not here to learn GPT pretraining (that's the payload — and it's curriculum item #3, "skip unless joining a foundation-model lab"). You're here for the harness: the loop, the metric, the spec file, and the guardrails. That structure is provider-, domain-, and task-agnostic. Master it once and you can wrap it around RAG tuning, prompt optimization, agent-policy search, feature engineering — anything with a number to push.

The mental model: hill-climbing with an LLM as the proposal engine

Every optimization loop has the same skeleton: propose a change → evaluate it → accept or reject → repeat. Classical methods (grid search, Bayesian optimization, evolutionary search) differ only in how they propose the next candidate. AutoResearch's insight is to make the proposal engine an LLM agent reading code and papers:

Role in the loopClassic optimizerAutoResearch
Propose next candidateAcquisition function / mutationAgent edits train.py using ML knowledge + prior results
EvaluateObjective functionTrain 5 min, compute val_bpb
Accept / rejectCompare to incumbentLower bpb → git commit; else → git reset
Search spaceFixed parameter vectorOpen-ended: any code edit

That last row is the qualitative leap. A hyperparameter sweep can only move within a box you defined in advance. An agent editing source code can change the architecture, swap the optimizer, add a regularizer it read about in a paper — an unbounded search space that no acquisition function can express. The price you pay for that freedom is exactly what the rest of the harness is built to control: an open-ended editor can also break the code, blow up memory, or quietly cheat the metric.

Part II

The Anatomy of a Harness

Forget GPUs for a moment. Abstract AutoResearch and you get five parts. This decomposition is the single most transferable thing in the course — when you build your own harness, you are filling in these five slots.

Immutable Substrateread-only
The fixed world the experiment runs in: data, the evaluation function, hard constants (time budget, sequence length). The agent cannot touch this. In the repo: prepare.py.
Mutable Surfaceagent edits this
The one place the agent is allowed to change. Everything inside is fair game. Keeping it to one file makes diffs reviewable and scope finite. In the repo: train.py.
The Spechuman writes this
Natural-language instructions: the rules, the goal, what's allowed, how to log, when to stop. This is the program you actually write — as prose, not code. In the repo: program.md.
The Metricsingle scalar
One number, lower-is-better, comparable across every candidate regardless of what changed. It lives inside the immutable substrate so the agent can't redefine its own grade. In the repo: val_bpb.
The Loopkeep / discard
The control flow that ties it together: propose, run, read metric, commit-or-revert, repeat forever. Implemented as agent behavior driven by the spec, with git as the accept/reject mechanism.

Why the immutable / mutable split is the whole game

Notice that two of the five parts (substrate, metric) exist specifically to constrain the agent, and the metric deliberately lives inside the substrate. This is not incidental — it is the core safety architecture:

Transferable principle #1

Separate what's measured from what's optimized, and make the measurer untouchable. If the agent that's trying to score well can also edit the scorer, it will eventually "improve" by editing the scorer — not the model. Every autonomous loop you ever build needs a bright line between the surface the agent optimizes and the harness that judges it. AutoResearch draws that line at the file level: train.py vs prepare.py.

As a data scientist this should feel familiar: it's the same instinct as never letting the model see the test set, or locking the eval harness before you start tuning. Most software engineers entering AI don't have that reflex — which, per your career notes, is part of your edge.

Part III

File by File

The repo has exactly three files that matter (prepare.py, train.py, program.md) plus two for results (results.tsv, analysis.ipynb). Open them alongside this section.

prepare.py — the immutable substrate

Run once. It downloads data shards, trains an 8,192-token BPE tokenizer, and — critically — defines the fixed constants and the evaluation function. The agent imports from it but never edits it.

# prepare.py — Constants (fixed, do not modify)
MAX_SEQ_LEN = 2048        # context length
TIME_BUDGET = 300         # 5-minute training budget, in seconds
EVAL_TOKENS = 40 * 524288 # tokens used for validation

VAL_SHARD = MAX_SHARD       # a PINNED validation shard — never trained on

Two design details worth absorbing:

  • The validation shard is pinned and excluded from training (the dataloader filters it out for the "train" split). This is the held-out set. It's enforced in code the agent can't change, so the agent literally cannot train on the eval data even if it "wanted" to.
  • The eval function carries a do-not-touch banner and is the sole source of truth for the metric:
# Evaluation (DO NOT CHANGE — this is the fixed metric)
@torch.no_grad()
def evaluate_bpb(model, tokenizer, batch_size):
    """Bits per byte: vocab-size-independent eval metric."""
    # ... sums per-token cross-entropy in nats, divides by total bytes ...
    return total_nats / (math.log(2) * total_bytes)
Note · enforcement is by convention here

The "do not modify" rule lives in program.md and a code comment — it's a social contract with a generally-obedient agent, not a hard filesystem lock. For higher-stakes setups, the transferable upgrade is to make it structurally impossible: run the eval in a separate process the agent can't edit, check the file's git hash before trusting a result, or sandbox file writes to the mutable surface only. Keep this gap in mind for Part VII.

train.py — the mutable surface

This is the only file the agent edits, and everything in it is fair game: the GPT architecture, the Muon+AdamW optimizer, the LR schedule, batch size, model depth. The human-facing knobs sit in a clearly marked block:

# Hyperparameters (edit these directly, no CLI flags needed)
ASPECT_RATIO     = 64        # model_dim = depth * ASPECT_RATIO
WINDOW_PATTERN   = "SSSL"    # sliding-window attention pattern
TOTAL_BATCH_SIZE = 2**19     # ~524K tokens per optimizer step
MATRIX_LR        = 0.04      # Muon LR for matrix params
DEPTH            = 8         # number of transformer layers (main size knob)
DEVICE_BATCH_SIZE= 128       # per-step batch (lower if OOM)

The training loop has one feature that's pure harness-design wisdom, independent of LLMs — it trains by wall-clock time, not by step count:

while True:
    # ... forward / backward / optimizer.step() ...
    progress = min(total_training_time / TIME_BUDGET, 1.0)
    # LR schedule, weight decay, etc. are all functions of `progress`
    if step > 10 and total_training_time >= TIME_BUDGET:
        break   # stop after exactly ~5 min of *steady-state* training

# Fast-fail guard: abort instantly if loss diverges
if math.isnan(train_loss_f) or train_loss_f > 100:
    print("FAIL"); exit(1)

Why time-budget rather than a fixed number of steps? Because the agent changes model size. A bigger model does fewer steps per second; comparing "1000 steps of a big model" against "1000 steps of a small model" is comparing different amounts of compute. Fixing wall-clock time makes every experiment cost the same compute, so val_bpb measures "best model achievable in 5 minutes on this hardware" — a genuinely comparable quantity. (The flip side: results aren't comparable across different hardware. Accepted tradeoff.)

program.md — the spec (the part you actually program)

This is the heart of the harness for your purposes. It's a Markdown "skill" the agent reads. It has five logical sections. Here is the actual structure, condensed:

## Setup
1. Agree on a run tag (e.g. `mar5`); branch `autoresearch/<tag>` must not exist
2. git checkout -b autoresearch/<tag>
3. Read README.md, prepare.py, train.py for full context
4. Verify data exists in ~/.cache/autoresearch/
5. Initialize results.tsv with just the header row
6. Confirm setup, then GO

## Experimentation — the rules
CAN:  modify train.py (architecture, optimizer, hyperparams, batch size...)
CANNOT: modify prepare.py · install new packages · touch the eval harness
GOAL: get the lowest val_bpb. Time budget is fixed, so ignore training time.
VRAM: a soft constraint — some growth ok for real gains, don't blow up.
SIMPLICITY: all else equal, simpler wins. A 0.001 gain from +20 hacky lines? skip.
            A 0.001 gain from DELETING code? definitely keep.
FIRST RUN: always establish the baseline (run train.py unchanged).

## Output format  → script prints `val_bpb: 0.997900` etc.; grep it from run.log

## Logging results  → append one row to results.tsv (TAB-separated):
   commit | val_bpb | memory_gb | status(keep/discard/crash) | description

## The experiment loop  → see Part IV
Transferable principle #2

The spec is your program; English is the language. Everything that makes the loop behave well — the goal, the constraints, the tie-breakers (simplicity!), the logging schema, the stopping rule — is encoded here, not in Python. Iterating on program.md is "programming your research org." This is where you'll spend your design effort when you adapt it. Note how much is value judgment, not mechanics: the simplicity criterion tells the agent how to trade quality against complexity — a policy you'd otherwise have to hard-code.

results.tsv + analysis.ipynb — the experiment ledger

Every run appends one tab-separated row. It's intentionally left untracked by git (it's a log of the search, not part of the code being searched):

commit   val_bpb    memory_gb  status   description
a1b2c3d  0.997900   44.0       keep     baseline
b2c3d4e  0.993200   44.2       keep     increase LR to 0.04
c3d4e5f  1.005000   44.0       discard  switch to GeLU activation
d4e5f6g  0.000000   0.0        crash    double model width (OOM)

analysis.ipynb then reads this ledger and computes the things you'd actually want at morning standup: the keep rate (how often a proposal stuck), the running-minimum frontier (best bpb over time), total improvement vs. baseline, and the top hits ranked by their delta. The delta for each kept experiment is measured against the previous kept state, because the search is cumulative — each accepted change builds on the last.

Why this matters more than it looks

A keep rate of, say, 3% (≈20 of 700) is not failure — it's normal for open-ended search, and it's exactly why you want an agent doing it overnight rather than a human doing it by hand. The ledger turns "the agent did stuff" into measured, auditable progress. When you build your own harness, design the log schema early: it's the difference between "I ran an autonomous loop" and "here is the frontier curve and the five changes that drove 80% of the gain."

Part IV

The Loop

The loop, step by step

From program.md's "experiment loop" section, this is the literal control flow the agent runs forever:

Look at git state — which branch/commit we're on (this is the current incumbent).
Edit train.py with one experimental idea (change the architecture, an LR, an optimizer detail…).
git commit the change.
Run it: uv run train.py > run.log 2>&1 — redirect all output to a file (never tee; see below).
Read the result: grep "^val_bpb:" run.log.
Empty grep ⇒ it crashed. tail -n 50 run.log for the traceback; fix-if-trivial, else give up on this idea.
Record the row in results.tsv (do not commit the tsv).
Improved (lower bpb)? → "advance the branch": keep the commit. The new state is the incumbent.
Equal or worse? → git reset back to where you started. The change is thrown away.

Git as the accept/reject state machine

There is no bespoke checkpoint manager, no experiment database, no orchestration framework. The "keep or discard" decision is git: commit to accept, reset to reject. The current HEAD is always the best-known configuration, and the commit history is a clean record of every accepted improvement.

Transferable principle #3

Use boring, battle-tested primitives as your control plane. Git already gives you atomic accept (commit), free rollback (reset), full audit history, and branching for parallel runs (autoresearch/mar5-gpu0, -gpu1…). Reaching for a heavyweight "agent framework" here would add failure modes without buying anything. The harness is ~3 files because it delegates state management to git and logging to a TSV.

Autonomy & context hygiene — the two non-obvious rules

Two instructions in program.md look like throwaway lines but encode hard-won agent-engineering lessons:

1 · "NEVER STOP"

The spec is emphatic: once the loop begins, the agent must not pause to ask "should I keep going?" or "is this a good stopping point?". The human is asleep. "If you run out of ideas, think harder — read the papers referenced in the code, re-read the files for new angles, combine previous near-misses, try more radical changes. The loop runs until the human interrupts you, period."

Why this is here

Left to default behavior, agents are sycophantically eager to check in — which destroys the entire value proposition of "run while I sleep." Genuine autonomy means you must explicitly grant it and then trust your guardrails (the discard mechanism) more than the agent's judgment about whether to continue. This is the inverse of "human-in-the-loop" (curriculum #34): you're deliberately designing the human out of the inner loop, which is only safe because every bad proposal is cheaply reverted.

2 · "Redirect everything — do NOT use tee or let output flood your context"

Training prints a fast-updating progress line every step. If that streamed into the agent's context window, a single 5-minute run could dump thousands of lines of noise, blowing the context budget and burying signal. So the output goes to run.log, and the agent reads back exactly two things via grep: the final val_bpb and peak VRAM. On crash, only tail -n 50.

Transferable principle #4

An autonomous agent's context window is a scarce resource you must actively manage (this is curriculum #33, Context Management, learned by example). The pattern: dump verbose tool output to disk, then pull back only the minimal scalar(s) you need to make the next decision. Over 100 iterations, the difference between "grep one line" and "stream the whole log" is the difference between an agent that runs all night and one that chokes on its own output after a dozen runs.

Part V

The Metric

val_bpb — bits per byte, and why not just "loss"

The metric is validation bits per byte: lower is better. Conceptually it's the model's cross-entropy on held-out text, but normalized per UTF-8 byte instead of per token:

$$\text{bpb} = \frac{\sum_i \text{CE}_i \;[\text{nats}]}{\ln(2)\,\cdot\,\sum_i \text{bytes}(t_i)}$$

The numerator sums the per-token cross-entropy (in nats); the denominator sums how many UTF-8 bytes each target token represents (special tokens, which have zero byte-length, drop out of both sums). Dividing by bytes and converting nats→bits via ln(2) gives bits per byte.

The crucial property: vocab-size independence

The agent is allowed to change the tokenizer's effective behavior and the model size, which changes how many tokens a given piece of text becomes. Plain "loss per token" would be unfair — a coarser tokenizer packs more bytes per token and looks artificially better. Normalizing per byte measures compression of the underlying text itself, so a 50M-param model and a 200M-param model with different token statistics are compared on the same footing. The metric was engineered to survive the very changes the agent is encouraged to make.

The immutable-eval keystone

We've now seen all three layers of metric protection, and they're worth stating together because this is the design pattern for any optimization-by-LLM loop:

  1. Held-out data — the validation shard is pinned and code-excluded from training. The agent can't fit the test set.
  2. Immutable scorer — evaluate_bpb lives in the file the agent can't edit. The agent can't redefine "good."
  3. Invariant metric — bpb is constructed to be comparable across the dimensions the agent varies. The agent can't win by exploiting an apples-to-oranges comparison.

Strip any one of these and the loop degrades into the agent gaming its own grade. Together, they make "lowest val_bpb" a goal that can only be reached by actually building a better model.

Part VI

Design Principles

The four choices that define the harness

ChoiceWhat it buysThe tradeoff
Single mutable file
(only train.py)		 Scope is finite and diffs are reviewable. The agent can't wander off into refactoring the data loader or the eval. Some ideas that would need substrate changes are off-limits by design.
Fixed time budget
(5 min wall-clock)		 Every experiment costs equal compute ⇒ results directly comparable; also finds the best model for your hardware in that budget. ~12 runs/hr, ~100/night. Results aren't comparable across different machines.
One scalar metric
(val_bpb)		 Unambiguous accept/reject. No multi-objective bargaining mid-loop. Things not in the metric (latency, robustness) are invisible unless you fold them in.
Self-contained
(no new deps, no distributed)		 Reproducible, debuggable, one GPU / one file / one metric. Fewer moving parts to break overnight. Caps the ceiling — no multi-node, no exotic libraries the agent might want.
The unifying theme

Every choice trades capability for comparability and safety. That's the right trade for an unattended loop: you'd rather have 100 trustworthy, comparable experiments than 100 fast-but-incomparable ones you can't reason about in the morning. Keep this lens when you design your own — the constraints are features.

Part VII

Guardrails & Where It's Still Weak

Goodhart's law and metric gaming

"When a measure becomes a target, it ceases to be a good measure." An agent optimizing a single number will exploit any loophole in how that number is computed. AutoResearch's defenses, mapped to the attack they stop:

Possible exploitWhat blocks it
Train on the validation dataVal shard pinned & code-excluded from the train split (immutable substrate)
Edit the eval to report a lower numberevaluate_bpb is in prepare.py, off-limits
Shrink vocab to make per-token loss look goodMetric is per-byte, vocab-independent
"Win" by overfitting a giant model5-min budget caps how much any model can memorize
Add unbounded complexity for a tiny gainSimplicity criterion in program.md tells the agent to reject it
Run forever / OOM the machine10-min hard timeout (kill & discard); VRAM soft cap; NaN fast-fail

Where the design is still weak (your DS opening)

The single-run noise problem

Each experiment is run once, and "improved" means the new val_bpb is lower — by any amount, including amounts smaller than run-to-run noise. With one seed and a stochastic 5-minute training run, a "0.0003 improvement" may be pure variance. Over 700 experiments you'll accept some changes that did nothing (false positives) and discard some that helped (false negatives). The original harness has no notion of statistical significance.

This is exactly where your background out-engineers the reference

A statistically-literate version of the keep/discard rule would: run each promising candidate over multiple seeds, estimate the noise floor, and only accept a change when the improvement clears it (a paired comparison, or a simple "Δ > k·σ" gate). That's a small change to the loop with an outsized credibility payoff — and a far stronger interview story than "I cloned autoresearch." Hold this thought for Part IX.

Other honest gaps: the immutable-eval rule is enforced socially (a cooperative agent), not by a sandbox; a single scalar hides regressions on anything not measured; and the agent can get stuck in local optima that a human would escape. None are fatal — they're the natural next iterations.

Part IX

Pointing It at Your Own Research Problem

This is the payoff. To wrap the harness around any problem, fill in the five slots from Part II. Here's the recipe, then a fully worked example on the RAG pipeline you already have in this repo.

The generalization recipe

SlotQuestion to answerFor a generic ML/LLM task
MetricWhat single number, lower-is-better, do I trust?Eval score from a frozen harness (RAGAS faithfulness, judge win-rate, accuracy, latency…)
Immutable substrateWhat must the agent never touch?The eval set, the scoring code, the test harness, hard resource/time limits
Mutable surfaceWhat single file/config can it edit?One config or one module: chunker params, prompt template, retriever settings, feature pipeline
SpecWhat are the rules, goal, tie-breakers, log format, stop rule?Your program.md — adapt the original's five sections

The one rule that makes or breaks it

Your eval must be fast, frozen, and a single scalar. If one evaluation takes an hour, you get ~8 experiments a night, not 100. If the eval set isn't frozen, the agent games it. If the metric is a vibe, the loop can't make decisions. Nail the metric first; everything else is plumbing you already understand.

Worked example: auto-tuning your RAG pipeline

You already have src/rag/ with chunking, embedding, retrieval, a CrossEncoder reranker, and — crucially — an eval layer (src/rag/eval/ragas_eval.py, judge.py). That eval layer is your evaluate_bpb. Here's the mapping:

AutoResearchYour RAG harness
prepare.py (substrate)A frozen eval set of (question, ground-truth) pairs + ragas_eval.py/judge.py, made read-only to the agent
train.py (surface)A single rag_config.py: chunk size/overlap, embedding model, top-k, reranker on/off, prompt template
5-min training runOne full pipeline run over the eval set (ingest if needed → retrieve → generate → score)
val_bpbA single composite score — e.g. 1 − mean(faithfulness, answer_relevancy, context_precision) so lower-is-better
git commit / resetidentical
Your upgrade over the reference (the DS move)

RAGAS/judge scores are noisier than a training loss — they depend on LLM sampling. So bake significance into the loop: evaluate each candidate over your full question set (and ideally 2–3 judge samples per answer), report mean and standard error, and define "improved" as the new mean beating the incumbent by more than the combined standard error — not just any decrease. This single change turns a noisy hill-climb into a defensible experiment, and it's the kind of rigor your career notes flag as your moat.

A starter program.md for the RAG harness

Drop something like this in a rag-autoresearch/ directory and point an agent at it. It's the original's five-section skeleton, retargeted:

# rag-autoresearch
Autonomously tune the RAG pipeline to maximize answer quality on a frozen eval set.

## Setup
1. Propose a run tag (today's date). Branch `rag/<tag>` must not exist.
2. git checkout -b rag/<tag>
3. Read: README, rag_config.py (you edit this), eval/run_eval.py (DO NOT EDIT).
4. Verify the frozen eval set exists at data/eval/qa_pairs.jsonl.
5. Create results.tsv with header. First run = baseline (config unchanged).

## Experimentation
CAN edit: rag_config.py only — chunk_size, chunk_overlap, embedding_model,
          top_k, use_reranker, rerank_top_n, prompt_template.
CANNOT: edit eval/, change the eval set, swap the judge model, add deps.
GOAL: minimize  score = 1 - mean(faithfulness, answer_relevancy, context_precision).
BUDGET: one eval pass = the full qa_pairs set. Cost is fixed; ignore wall time.
SIGNIFICANCE: run eval, record mean AND std error over the question set.
   "improved" = new_mean < best_mean - (se_new + se_best).   # beat the noise

Run:  python run_eval.py --config rag_config.py > run.log 2>&1
Read: grep "^score:\|^score_se:" run.log     # keep context clean
Append to results.tsv (TAB-sep):
  commit | score | score_se | status(keep/discard/crash) | description

1. note git state  2. edit rag_config.py  3. git commit  4. run eval
5. grep score  6. crash? tail -50, fix-or-skip  7. log row

Part X

Reference

Design checklist for any autonomous eval loop

  • ☐ One scalar metric, lower-is-better, computed by a frozen scorer.
  • ☐ Held-out eval data, structurally excluded from anything the agent can influence.
  • ☐ Bright line between the mutable surface (what's optimized) and the immutable substrate (what judges).
  • ☐ Fixed cost per experiment (time/compute/tokens) so candidates are comparable.
  • ☐ Fast eval — aim for >10 experiments/hour, or the loop isn't worth automating.
  • ☐ Cheap accept/reject — git commit/reset; no bespoke state store.
  • ☐ A logged ledger with a defined schema; analysis notebook for the frontier curve.
  • ☐ Explicit autonomy grant ("never stop") plus hard safety rails (timeout, resource cap, fast-fail).
  • ☐ Context hygiene — verbose output to disk, grep back only the scalar(s).
  • ☐ A tie-breaker policy (e.g. simplicity) so the agent doesn't accumulate junk for noise-level gains.
  • ☐ (Your edge) A significance gate so "improved" means "beat the noise," not "moved the number."

FrontierSWE: Reading Proximal Labs'
Long-Horizon Coding Benchmark

FrontierSWE grades coding agents on tasks that take hours, not minutes: rewrite a compiler backend, build a lossless notebook compressor, train a molecular-graph regressor under a closed-data cap. This doc reads the cloned repo end to end — task spec, agent loop, verifier, anti-cheat — then lines it up next to autoresearch to see what a "small, honest harness" and a "production benchmark harness" actually share.

Repo: Proximal-Labs/frontier-swe 17 tasks, ~400 commits Built on the Harbor framework
Part I

The Big Idea

What FrontierSWE actually is

FrontierSWE is a benchmark from Proximal Labs that tests coding agents on ultra-long-horizon problems — the kind that would occupy a strong engineer or researcher for hours to days, not a single diff. The first release ships 17 tasks split across three categories: implementation (port a real system to a new language — e.g. git-to-zig, lua-native-compiler), performance (rewrite a hot path for speed without breaking correctness — e.g. ffmpeg-swscale-rewrite, cranelift-codegen-opt), and ml_research (open-ended modeling under a constraint — e.g. pcqm4mv2-autoresearch, frogsgame-rl). Results are tracked on a public leaderboard at frontierswe.com.

Because the tasks are this large, binary pass/fail grading doesn't make sense — a task can be "70% correct and 1.4x faster" and that's a meaningful, comparable outcome. So every task is graded on a continuous [0,1] scale instead, via a per-task custom verifier.

It doesn't reinvent the harness — it sits on Harbor

The single most important structural fact about this repo: FrontierSWE is not a standalone harness. It's a content + glue package built on top of Harbor, an open framework (from the team behind Terminal-Bench) for specifying and running sandboxed agent evaluations. The frontier-swe repo imports harbor as a dependency and supplies three things Harbor doesn't ship with: the tasks (problem content + verifiers), harbor_ext (per-agent-CLI shims), and scoring (the leaderboard-specific reward math).

Why this matters for the comparison

AutoResearch is a single, self-contained repo that is its own harness. FrontierSWE's repo is the content layer on a separately-versioned generic orchestration framework. Reading it well means reading two boundaries: where Harbor's job ends and harbor_ext/tasks begins, and where the agent's writable surface ends and the verifier's authority begins.

Part II

Harness Anatomy

Three layers, cleanly separated by directory

Harbor (pip dep)not in this repo
Generic trial orchestration: spins sandboxes, runs an agent against an environment for a budget, collects logs/artifacts, calls the verifier, writes reward.json. Knows nothing about notebooks, compilers, or molecules.
harbor_ext/agent shims
9 Python files, one per coding-agent CLI (claude_code.py, codex.py, gemini_cli.py, qwen_code.py, kimi_cli.py, opencode_cli.py, cursor_cli.py, deepseek_claude.py, glm_claude.py) — translates "run this instruction" into the right CLI invocation, env vars, and auth for that specific tool, plus shared safety helpers (agent_shell_safety.py, network_allowlist.py, modal_managed.py).

Part III

A Task, File by File

Take tasks/notebook-compression/ as the worked example: build a lossless, domain-specific compressor for .ipynb files, graded against a hidden holdout corpus.

task.toml + job.yaml — the immutable resource contract

task.toml is short and load-bearing: it fixes the agent's wall-clock budget (agent.timeout_sec = 28800.0, 8h), the verifier's own budget (verifier.timeout_sec = 14400.0, 4h), the sandbox's resources (16 vCPU, 32 GiB RAM, 150 GiB scratch, no GPU, no internet), and which pinned Docker image to boot (ghcr.io/proximal-labs/frontier-swe/notebook-compression:v5). job.yaml is the orchestration-side sibling: it names the run (notebook-compression-full-sweep), sets n_attempts: 3 trials, picks the environment backend (harbor_ext.modal_managed:ManagedModalEnvironment), and lists every agent+model pair to run against this one task — each with its own override_timeout_sec, which can exceed task.toml's default.

A live wrinkle, in the repo's own comments

job.yaml's artifacts list carries a comment explaining why /app (the whole working directory, not just the declared output paths) is captured: a prior model run left helper modules outside the declared submission paths and scored zero. The fix applied was widening artifact capture, not editing instruction.md — because changing the task spec mid-benchmark would make already-scored trials incomparable to future ones. This is the same "immutable substrate" instinct as AutoResearch's read-only prepare.py, but FrontierSWE has to defend it against its own need to patch bugs without invalidating a public leaderboard.

instruction.md — the spec the agent reads

This is FrontierSWE's analogue of program.md, but task-specific and far more contractual. For notebook-compression it defines: the exact metric formula (a geometric-mean compressed/original byte ratio over hidden notebooks), the submission contract (a single executable /app/run with fit / compress / decompress subcommands), what must live where (everything decompress needs must be inside artifact_dir, because decompress runs in a fresh sandbox with only that directory and the compressed input), hard resource caps per stage, and a "Behavioral Rules" section that is — almost verbatim — AutoResearch's autonomy mandate:

# from tasks/notebook-compression/instruction.md
- Never stop to ask. Work autonomously until interrupted.
- Check time regularly with `cat /app/.timer/remaining_secs`.
- Keep `/app/run` valid and executable at all times.
- Keep a self-eval result in `/app/dev_results/` so you can track progress.
- Optimize for the hidden holdout, not for pathological compression
  of your own validation split.

tests/ — the verifier, sealed off from the agent

The agent never sees tests/hidden_test_set_bundle.zip. tests/test.sh is the Harbor-facing entrypoint: it logs to /logs/verifier/verifier.log, extracts the hidden bundle into a temp dir, runs the task's own scoring code (compute_reward.py / scoring_core.py), and — critically — has a trap that writes a fallback failure reward.json if anything goes wrong, so a crash never silently produces "no score" (it produces a guaranteed worst-case score instead).

Part IV

The Agent Loop

The timer daemon — fixed budget, agent-readable

Where AutoResearch hard-codes a 5-minute training run inside train.py itself, FrontierSWE enforces the wall-clock budget outside the agent (via Harbor's trial timeout) and exposes it back in as plain files the agent is told to poll:

cat /app/.timer/remaining_secs   # seconds remaining
cat /app/.timer/elapsed_secs     # seconds elapsed
test -f /app/.timer/alert_30min  # true when <=30 min remain
test -f /app/.timer/alert_10min  # true when <=10 min remain

Every instruction.md repeats the same shape of inner loop. pcqm4mv2-autoresearch's is the clearest, and it's worth reading next to AutoResearch's own loop (Part VII has the side-by-side):

Edit train.py or supporting code.
Run python3 train.py > run.log 2>&1.
Check grep "dev_mae" run.log.
Keep the best checkpoint; regenerate predict.py if needed.
Repeat until time expires — maintain valid artifacts even on crash.

claude_code.py — what actually happens when the agent runs

harbor_ext/claude_code.py shows the literal exec path for one agent. It is far more involved than "spin up Claude in this repo": it wires ANTHROPIC_API_KEY (with Z.ai-compatible fallbacks), forces --permission-mode=bypassPermissions (no interactive approval — the trial must run unattended), explicitly passes --disallowed-tools WebSearch WebFetch (search is disabled so the benchmark measures the model's own capability, not its ability to find the answer online), and — notably — has a built-in one-shot recovery path: if the run dies from exceeding the output-token limit, it greps the crashed session's ID out of the log and resumes that exact session with a hint to write smaller chunks, with no second retry.

Transferable idea

The "websearch/webfetch disabled for fairness" choice is the same instinct as AutoResearch's "no new dependencies" rule — both are guarding against the agent improving its score by reaching outside the intended search space, rather than by getting better at the actual problem.

Sandbox safety wrappers — hiding the host from the agent

agent_shell_safety.py installs replacement pgrep/pkill binaries ahead of the real ones on PATH, that filter out the calling process's own ancestor chain. An autonomous agent running unattended for 8+ hours will routinely try to inspect or kill processes; this stops it from seeing or signaling Harbor's own supervising processes, without the agent ever being told this wrapper exists. It's a small, concrete example of a class of guardrail AutoResearch doesn't need at all — because AutoResearch runs on the user's own machine, in the user's own terminal, with no host process to hide.

Part V

The Eval

reward.json — what the verifier actually emits

Every verifier writes a structured reward.json: a top-level reward/score, plus component fields (subscores, additional_data — per-workload pass counts, measured speedups, module breakdowns). SCORING.md is explicit that the top-level score in the raw verifier output is conservative — often hard-zeroed on any correctness miss — and that the leaderboard's real, partial-credit score is derived from the components by a second pass.

Category-gated scoring — score_from_reward.py

This is the piece that turns 17 wildly different metrics (compression ratio, MAE, test-pass-rate, board-clears) into one comparable [0,1] number, by category:

CategoryGated score formulaIntuition
implementationcorrectnessPure pass-rate against hidden tests.
performance0.5 * correctness, or 0.5 + 0.5 * speedup once correctness == 1.0Speed only counts once correctness is perfect — you cannot trade correctness for speed.
ml_researchraw reward (e.g. frogsgame-rl: board-clear count / 500)No gate; the task's own metric is the score.
notebook-compression (special)speedup-equivalent (compression ratio) if fully correct, else 0Round-trip exactness is binary — any byte mismatch is a hard fail ranked below every valid run.
libexpat-to-x86asm (special)weighted per-module correctness gates an uncapped performance scorePartial credit across 7 weighted test modules before performance counts at all.

The leaderboard then aggregates 5 trials per (task, agent) pair into Avg Score and Best Score, plus a Correctness X/5 column counting how many trials reached 100% correctness (i.e. unlocked the speedup tier) — which can be 0/5 while Avg/Best are still non-zero, since partial correctness still earns partial credit on the correctness-only side of the gate.

Anti-cheat and hidden data

Two structurally separate defenses, doing different jobs: hidden test data (the agent's sandbox never contains tests/hidden_test_set_bundle.zip or its unpacked contents — it can only see the visible/dev split) stops the agent from directly reading the answer; a separate post-hoc anti-cheat audit (scoring/anticheat.json) flags trials after the fact and zeroes them regardless of their computed score — catching things like exfiltrating verifier internals through side channels, which hiding files alone can't prevent.

Part VI

Orchestration & Infra

Modal sandboxes — ManagedModalEnvironment

harbor_ext/modal_managed.py extends Harbor's generic Modal backend with FrontierSWE-specific plumbing: domain/CIDR network allowlists (built from live AWS/Google IP-range feeds plus per-agent fallback domain lists, since most tasks set allow_internet = false and only the agent CLI's own API endpoints are allowed through), GHCR-authenticated image builds (build_registry_token_env), persisted trial-state volumes so long sandboxes survive interruptions, and debug probes that periodically sample disk usage and tail agent logs — observability that matters when a single trial can run unattended for 8+ hours in the cloud.

The multi-agent matrix — this is a leaderboard, not a session

A single job.yaml fans one task out across many agent CLIs and models at once — Claude Code on Opus, Codex on GPT-5.4, Gemini CLI, Qwen Code, Kimi CLI, OpenCode on GLM-5, each with its own reasoning-effort/thinking knobs — so the same frozen task and verifier produce a directly comparable row per model on the public leaderboard. This cross-agent fan-out is the single biggest structural difference from AutoResearch, which assumes exactly one agent, in one repo, on one machine, with no comparison being made to anyone else's run.

Part VII

Side by Side with AutoResearch

The literal lineage: pcqm4mv2-autoresearch

Not just a similar idea — a named homage

One of FrontierSWE's three ml_research tasks is literally called pcqm4mv2-autoresearch, and its instruction.md "Experiment Loop" section (quoted in Part IV) reproduces karpathy's autoresearch pattern almost verbatim: edit the one mutable file, run it, grep one metric out of a log, keep the checkpoint if it improved, repeat until time runs out, never stop to ask. FrontierSWE didn't just converge on the same idea independently — it embedded AutoResearch's exact loop shape as one task inside a much larger benchmark, then wrapped it in hidden-eval, multi-trial, multi-agent machinery that AutoResearch itself doesn't have.

Part VIII

Design Principles

What each harness is actually optimizing for

Both harnesses converge on the same core invariant — a hard line between what the agent may edit and what judges it — but they spend that invariant on different goals.

  • AutoResearch optimizes for replicability on one machine. Every design choice (fixed 5-minute runs, one file to edit, git as the only state store, no cloud dependency) exists to make the loop runnable overnight on a single GPU you already own, with nothing else to configure.
  • FrontierSWE optimizes for cross-agent comparability at scale. Category-gated scoring, hidden holdout data, network allowlists, anti-cheat audits, and pinned Docker images all exist to make sure that when Claude Opus scores 0.71 and GPT-5.4 scores 0.68 on the same task, that 0.03 gap reflects a real capability difference and not a sandboxing leak or a frozen-instruction drift between trials run weeks apart.
  • Both treat the agent loop itself as a known, reusable shape. Propose → run → measure-against-a-frozen-metric → keep-or-discard → repeat-until-budget-exhausted appears almost unchanged in both repos — it's evidence this is close to a canonical pattern for any autonomous optimization loop, independent of scale.
Part IX

Where Each Is Still Weak

FrontierSWE's open seams

Two sources of truth for the score

SCORING.md states outright that the raw reward.json["score"] is conservative and doesn't match the published leaderboard score — a second script (score_from_reward.py) has to re-derive it from components. The doc itself flags this as temporary ("the next release will emit the gated score directly"), but until then, anyone reproducing scores from raw verifier output without that second pass will get numbers that don't match the leaderboard.

Anti-cheat is post-hoc, not preventive

A trial can run to completion, score well, and only later get zeroed by the anti-cheat audit. That's a reasonable trade given how much surface an 8-hour, internet-adjacent, unattended agent session has — but it means the harness trusts after-the-fact detection over architectural prevention for whatever the audit catches that hidden-data and network allowlisting don't.

Patching around the spec instead of fixing it

The notebook-compression job.yaml comment (Part III) is a real instance of the immutability-vs-fairness tension: a bug in how artifacts were captured zeroed valid trials, and the fix was widening artifact capture rather than clarifying instruction.md, specifically to avoid changing the task for future models versus already-scored ones. That's defensible, but it means the spec the agent reads and the spec the verifier actually enforces can quietly diverge over the benchmark's lifetime.

Nothing here is runnable on a laptop

Every task needs a Modal account, GHCR image pulls, and API keys for whichever agent CLI — there's no equivalent of AutoResearch's "single GPU, uv sync, go." FrontierSWE is built to be operated by Proximal Labs as a benchmark service, not picked up and pointed at your own problem the way AutoResearch's README explicitly invites.

AutoResearch's open seams, from the other side

No hidden eval, no anti-gaming beyond instruction-following

FrontierSWE's hidden-data + network-allowlist + anti-cheat stack exists because, at multi-hour unattended scale, an agent will find shortcuts a human reviewer might miss. AutoResearch's single keystone ("don't touch prepare.py", enforced by a human glancing at diffs) works at 5-minute-loop, single-user scale — but it has no answer to an agent that, say, overfits to the validation split or quietly increases its own time budget, the way FrontierSWE's holdout/allowlist design is built to catch.

No cross-run comparability

AutoResearch's README says this directly: results aren't comparable across different people's compute. FrontierSWE's entire raison d'être — pinned images, fixed resource envelopes, multi-trial averaging — is the answer to exactly that gap, at the cost of all the infra weight in Part VI.

Part X

Reference

A transferable checklist, updated for benchmark-scale loops

Extending the single-user checklist from the AutoResearch crash course with what FrontierSWE adds once you need cross-agent, cross-run comparability:

  • ☐ Hidden eval data the agent's sandbox structurally cannot reach, not just an instruction not to look.
  • ☐ Pinned, versioned environments (container images) so "the same task" actually means the same task months apart.
  • ☐ A category-aware scoring function when raw outcomes aren't comparable across task types (correctness-gates-speedup, raw-reward-only, etc.).
  • ☐ A fallback score on crash/timeout so a verifier failure never produces a missing data point.
  • ☐ Network/process isolation proportional to how long and how unattended the agent runs.
  • ☐ A post-hoc audit layer as a backstop for whatever the architectural defenses miss.
  • ☐ A documented, standalone scorer (like score_from_reward.py) so the leaderboard math isn't a black box.

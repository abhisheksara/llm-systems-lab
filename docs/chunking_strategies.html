Strategy 01

Fixed-Size
Chunking

The simplest and most widely used strategy. Text is divided into chunks of a predetermined size — measured in characters or tokens — with an optional sliding overlap window to preserve context at boundaries.

A chunk size of 512–1024 tokens with 10–20% overlap is a common baseline. The overlap ensures that information spanning a chunk boundary isn't lost during retrieval.

Strengths
  • Trivial to implement
  • Predictable chunk count
  • Near-zero compute cost
  • Works for any modality
  • Easy to tune overlap
Weaknesses
  • Ignores semantic boundaries
  • Splits mid-sentence often
  • Overlap inflates index size
  • Uniform granularity mismatch
When to use Prototyping, homogeneous corpora (product catalogues, logs), or when latency and cost dominate quality requirements. Establish as a baseline before investing in smarter strategies.
Python 
def fixed_chunk(text: str, size: int = 512, overlap: int = 64) -> list[str]:
    chunks = []
    start = 0
    while start < len(text):
        end = start + size
        chunks.append(text[start:end])
        start += size - overlap  # slide window with overlap
    return chunks

# LangChain equivalent
from langchain.text_splitter import CharacterTextSplitter

splitter = CharacterTextSplitter(
    chunk_size=512,
    chunk_overlap=64,
    separator="",  # no separator = pure fixed split
)
chunks = splitter.split_text(document)

Strategy 02

Recursive Text
Splitting

A smarter fixed-size variant that respects document structure. It tries a hierarchy of separators — paragraphs, then newlines, then sentences, then words — splitting only when a chunk exceeds the target size.

This is the LangChain default and the de facto production standard. It produces chunks that almost always end at natural linguistic boundaries while still guaranteeing a maximum size.

Strengths
  • Respects linguistic structure
  • Rarely splits mid-sentence
  • Configurable separator hierarchy
  • Still O(n) compute cost
  • Production-battle-tested
Weaknesses
  • Chunk sizes vary
  • Separator list needs tuning
  • Doesn't understand meaning
  • Topic changes still get mixed
When to use General-purpose RAG pipelines, structured documents (markdown, code, reports). This should be your default when you don't have the budget or latency budget for embedding-based splitting.
Python 
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,
    chunk_overlap=100,
    separators=[
        "\n\n",   # paragraph break (try first)
        "\n",     # newline
        ". ",     # sentence end
        " ",      # word boundary
        "",       # character (last resort)
    ],
)

chunks = splitter.split_text(document)

# For code: language-aware separators
from langchain.text_splitter import Language
code_splitter = RecursiveCharacterTextSplitter.from_language(
    language=Language.PYTHON,
    chunk_size=2000,
    chunk_overlap=200,
)

03
Strategy 03

Semantic
Chunking

The most sophisticated approach. Every sentence is embedded using a language model and cosine similarity is computed between adjacent sentences. When similarity drops below a threshold, a chunk boundary is placed.

The result: chunks that capture a single coherent topic or thought, regardless of their token length. This maximally aligns with how vector search retrieves — by meaning, not position.

Strengths
  • Meaning-preserving boundaries
  • Best retrieval precision
  • Topic-coherent chunks
  • Adapts to document structure
Weaknesses
  • Requires embedding at index time
  • 2–10× higher ingestion cost
  • Threshold is hard to tune
  • Chunks vary wildly in size
  • LLM context limits can break
When to use High-stakes retrieval where precision matters: legal documents, medical literature, research papers. When retrieval quality directly impacts downstream accuracy and you can absorb the embedding cost at ingest time.
Python 
from langchain_experimental.text_splitter import SemanticChunker
from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

chunker = SemanticChunker(
    embeddings=embeddings,
    breakpoint_threshold_type="percentile",  # or "std_dev", "interquartile"
    breakpoint_threshold_amount=95,          # split at top 5% similarity drops
)

chunks = chunker.split_text(document)

# From scratch: compute cosine drops manually
import numpy as np

def cosine_sim(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

# Embed sentences, compute adjacent similarities, split on drops
sentences = text.split(". ")
embeds = embeddings.embed_documents(sentences)
sims = [cosine_sim(embeds[i], embeds[i+1]) for i in range(len(embeds)-1)]
threshold = np.percentile(sims, 5)  # bottom 5% = split points

Side-by-Side Comparison

Strategy Boundary Quality Compute Cost Complexity Chunk Size Best For
Fixed-Size Poor Negligible Simple Uniform Prototypes, logs, uniform text
Recursive Splitting Good Negligible Moderate Near-uniform Production default, markdown, code
Semantic Chunking Excellent High (embeddings) Complex Variable High-precision RAG, research docs
Rule of Thumb

Start with Recursive Splitting. Move to Semantic only after measuring retrieval quality and confirming the embedding cost is justified by recall gains.

Tuning Chunk Size

Match chunk size to your LLM's context window and query complexity. Short factual queries → smaller chunks (256–512). Long analytical queries → larger chunks (1024–2048).

Evaluating Quality

Use RAGAS metrics: context precision and context recall tell you how well your chunks serve retrieval before touching generation quality.
